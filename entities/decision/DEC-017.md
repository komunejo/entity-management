Ordinary Markdown is the default link syntax for a reference in a record's prose, and the default form is the target's title linked to its path, with the ID riding behind it linked to the same path:

```
[entities are referenced by stable IDs with a unique prefix per type](DEC-004.md)^[DEC-004](DEC-004.md)
```

A project that prefers wikilinks declares it. Nothing is assumed on its behalf, which is how this engine already treats its other contracts — [per-project contract policy, asked explicitly at first-schema creation](DEC-007.md)^[DEC-007](DEC-007.md) asks the owner to choose the contract policy rather than choosing for them.

The reason for the default is dependency, not taste. A wikilink obliges every reader of a project to be a wiki, and a registry whose whole claim is that its records stay legible in any text editor ([documents must remain human-readable and human-editable](../requirement/REQ-002.md)^[REQ-002](../requirement/REQ-002.md)) cannot ship, as its default, a syntax that only one editor understands — and, as [the references the engine validates are dead links in every reader](../issue/ISSUE-006.md)^[ISSUE-006](../issue/ISSUE-006.md) measures, currently not even that one.

The label is the target's `title` rather than free text, and the ID rides alongside rather than replacing it, because that makes every part of a reference a function of the ID: label, path and annotation are all recoverable from it, so the whole construct can be verified and regenerated. The title, and not the filename: a filename is a slug at best and the bare ID at worst, whereas the title is the prose a person would have written anyway. It is also the field the generated index already labels its rows with, so one rule serves both — and [configurable display field for index generation](../proposal/PROP-002.md)^[PROP-002](../proposal/PROP-002.md) is the same rule meeting a project whose entities carry no title at all, which is where it will have to become declarable. That derivability is what closes the hole a plain Markdown link leaves open. A plain link is checkable but not repairable — resolve a stale path and you learn that the reference is broken without learning what it meant. The ID survives the rename the path does not, which is [entities are referenced by stable IDs with a unique prefix per type](DEC-004.md)^[DEC-004](DEC-004.md)'s own argument arriving where it was always needed: the reference becomes maintainable by the engine instead of by whoever remembers, which is what [hard integrity must not depend on model discipline](../requirement/REQ-004.md)^[REQ-004](../requirement/REQ-004.md) demands and what a path sitting inside a paragraph has never had.

Measured rather than assumed, on the renderer this repository is published through. Both links render and both open. The caret stays a literal caret — there is no inline footnote syntax — which reads as the footnote mark it imitates; `<sup>` would raise it properly at the price of putting HTML in the prose, and raw legibility is the requirement, so the caret stands. One trap earns its place here because it fails the same silent way this decision exists to stop: a destination containing spaces kills the link and renders as plain text. Angle brackets around it fix that and leave the raw path readable, where percent-encoding fixes it and does not. If filenames carry spaces, the destination is angle-bracketed — and nobody writes that by hand, which is the argument for the engine writing it.

The engine checks the form: the ID resolves, the two destinations agree, and they point at that ID's file, relative to the record making the reference. It does not read the label. A wiki-style link is still accepted and still checked for its ID, for projects whose readers understand one.

The label is free on purpose, and that places it on the soft side of [hard integrity is enforced by the engine; soft integrity is agent-assisted review](DEC-006.md)^[DEC-006](DEC-006.md): a stale label is a contradiction inside prose — review rather than guarantee, and the responsibility of whoever wrote it. It could have been otherwise, since a label is derivable from the ID and an engine could bind it. Leaving it free is what buys *that decision* as a label instead of a recital of the title, and the price is that a changed title leaves every label quoting it behind with nothing noticing.

A project for which that trade runs the other way asks its agents to check labels, and that is where such a rule belongs: a soft check lives in the instructions an agent reads, the way a hard one lives in the schema the engine reads. Binding the label in the engine instead would mean letting it write inside a record's body rather than only into the index ([engine writes preserve a file's line endings, and default to LF](DEC-014.md)^[DEC-014](DEC-014.md)) — a boundary [generated views inside records: reverse-reference tables](../proposal/PROP-003.md)^[PROP-003](../proposal/PROP-003.md) is already pushing at from the other side.
