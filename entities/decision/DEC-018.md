title: a filename is an ID and an optional handle; the name is the title
status: accepted
date: 2026-07-15
tags: [architecture, references, schema-language]

Three things with three jobs, and confusing them has cost this project a day. The **ID** is identity: `NS-###`, allocated once, never renamed, and the only thing a reference resolves through ([entities are referenced by stable IDs with a unique prefix per type](DEC-004.md)^[DEC-004](DEC-004.md)). The **title** is the name: a field, and the label a reference shows. A **slug** is neither — it is an optional handle on the filename, chosen by hand and derived from nothing.

The engine's rule is that a filename starts with its ID; what follows is the handle, and it is optional. This project's default is that nothing follows: `DEC-004.md`. Short names are faster to operate — less to type, less to read down a listing — and that is a preference, not a finding. A default belongs to whoever leads the project and holds until they say otherwise; the pattern itself belongs to each project, which is why it is the project's to declare and not the engine's to fix.

A project that wants its filenames to read declares so and keeps the name in them. A `title` field would then be a second copy of that name, so such a project has none — an exception someone chose, never a default to drift into. With nothing declared, records have titles. Where the declaration lives is the schema, alongside `id`, `dir` and `path` — not `CLAUDE.md`, which the engine never opens, and where a pattern would be honoured only by an agent remembering to honour it while the engine went on rejecting every record.

Nothing derives the handle from the title, and nothing should. `new` used to slugify the title into the filename, and the rename workflow said keeping that copy current was optional — so every record here carried a hand-written paraphrase of its own name that no code read and nothing checked. Removing them costs nothing: references resolve by ID and never look at names.
